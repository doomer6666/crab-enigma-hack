# Jarvis AI

Проект для хакатона: AI-агент, который автоматизирует обработку входящих писем техподдержки.

---

## Суть решения

Система забирает письма из почтового ящика техподдержки, прогоняет через AI (классификация + генерация ответа), показывает оператору результат в виде веб-таблицы. Оператор смотрит, правит если надо, жмёт «Отправить» — ответ уходит клиенту. Всё.

---

## 3. Идейная реализация решения

### 3.1 О чём проект

Оператор техподдержки получает десятки писем в день. Читать каждое, понимать суть, искать решение в базе знаний, писать ответ — это долго и монотонно. JARVIS берёт на себя рутину: сам забирает письма, понимает о чём они, находит подходящее решение и пишет черновик ответа. Оператору остаётся только проверить, поправить если надо и нажать «Отправить».

### 3.2 Пользовательский путь

Описываем от лица оператора техподдержки — что он делает шаг за шагом.

**Шаг 1. Открывает систему**
Оператор заходит в веб-интерфейс JARVIS. Видит дашборд с основными цифрами: сколько новых обращений, сколько критических, сколько уже обработано AI. Ниже — таблица со всеми обращениями.

**Шаг 2. Смотрит таблицу обращений**
Таблица показывает все входящие письма. Каждое письмо уже автоматически обработано AI-агентом. В таблице видно:

- Тема письма
- Кто написал
- Категория (определил AI — например «Авторизация», «Оплата», «Техническая проблема»)
- Приоритет (AI оценил по содержанию — low / medium / high / critical)
- Тональность (позитивная, нейтральная, негативная)
- Процент уверенности AI в своей классификации
- Статус обращения

Оператор может фильтровать по статусу, приоритету, искать по тексту. Новые и критические обращения подсвечиваются.

**Шаг 3. Открывает конкретное обращение**
Кликает на строку в таблице. Справа выезжает панель с деталями:

- Полный текст письма клиента
- Вся цепочка переписки (если клиент писал не первый раз)
- Метаданные от AI: категория, приоритет, тональность, уверенность
- Черновик ответа, который подготовил AI

**Шаг 4. Работает с ответом**
Оператор видит кнопку «Вставить AI-черновик». Нажимает — в поле ответа появляется текст, который сгенерировал AI на основе письма клиента и базы знаний. Оператор может:

- Использовать черновик как есть
- Отредактировать его
- Написать свой ответ с нуля

**Шаг 5. Отправляет ответ**
Нажимает «Отправить». Ответ уходит клиенту на email через SMTP. Тикет автоматически переходит в статус «Решён». Исходящее сообщение сохраняется в истории переписки.

**Шаг 6. Экспорт данных**
В любой момент оператор может нажать «Экспорт» и скачать таблицу всех обращений в формате XLSX. Удобно для отчётности и аналитики.

Сводная таблица пользовательского пути:

| Шаг | Действие оператора              | Что происходит в системе                                    |
| --- | ------------------------------- | ----------------------------------------------------------- |
| 1   | Открывает веб-интерфейс         | Загружается дашборд и таблица                               |
| 2   | Просматривает таблицу           | Данные подтягиваются из БД, обновляются каждые 5 сек        |
| 3   | Кликает на обращение            | Загружаются детали тикета и история переписки               |
| 4   | Нажимает «Вставить AI-черновик» | Черновик из БД подставляется в редактор                     |
| 5   | Редактирует и отправляет ответ  | Backend отправляет email через SMTP, обновляет статус в БД  |
| 6   | Нажимает «Экспорт»              | Backend формирует XLSX из данных БД, браузер скачивает файл |

### 3.3 Архитектура системы

Система состоит из четырёх сервисов. Все поднимаются через docker-compose.

#### Frontend (React + TypeScript)

Веб-интерфейс для оператора. Тёмная тема в стиле JARVIS. Что внутри:

- Дашборд со статистикой (количество по статусам, приоритетам, тональностям)
- Таблица обращений с фильтрами и поиском
- Боковая панель с деталями тикета и перепиской
- Редактор ответа с поддержкой AI-черновиков
- Экспорт в XLSX
- Страница базы знаний с поиском

Фронт общается только с бэкендом через REST API. Никаких прямых вызовов к ML-сервису или базе данных.

#### Backend (FastAPI, Python)

Центральный сервис, через который проходит всё. Его задачи:

**Получение почты.** Фоновая задача каждые 30 секунд опрашивает почтовый ящик техподдержки через IMAP. Забирает новые письма, парсит тему, отправителя, тело письма.

**Вызов AI-агента.** Когда приходит новое письмо, бэкенд отправляет его текст в ML-сервис. Получает обратно: категорию обращения, приоритет, тональность и степень уверенности. Затем отправляет второй запрос — на генерацию черновика ответа (вместе с контекстом из базы знаний).

**Работа с базой знаний.** Перед генерацией ответа бэкенд ищет релевантные статьи в базе знаний. Сейчас это текстовый поиск по ключевым словам (ILIKE). На хакатоне планируем добавить векторный поиск через pgvector или embeddings.

**Заполнение веб-таблицы.** Тикет со всей классификацией и AI-черновиком сохраняется в PostgreSQL. Фронт подтягивает данные через REST API. Таблица обновляется автоматически — оператор видит новые обращения без перезагрузки страницы.

**Отправка ответа.** Когда оператор нажимает «Отправить», бэкенд берёт текст, отправляет через SMTP на email клиента, сохраняет исходящее сообщение в БД и меняет статус тикета.

**REST API для фронта.** Все эндпоинты для CRUD тикетов, получения переписки, поиска по базе знаний, экспорта данных.

#### ML Service (FastAPI + Transformers)

Отдельный микросервис с AI-моделями. Вынесен отдельно по двум причинам: модели тяжёлые (жрут память), и чтобы можно было менять/обновлять модели независимо от основного бэкенда.

Два эндпоинта:

- `POST /classify` — принимает текст письма, возвращает категорию, приоритет, тональность и confidence. Сейчас работает на rule-based логике (ключевые слова). На хакатоне подключим BERT/RoBERTa.
- `POST /generate` — принимает текст письма и контекст из базы знаний, возвращает черновик ответа. Сейчас подставляет шаблон по категории. На хакатоне — RAG или fine-tuned модель.

Rule-based классификатор — это не заглушка, а осознанный fallback. Если ML-модель не успеем дотюнить или она будет давать плохое качество, rule-based подход продолжит работать.

#### PostgreSQL

Хранит все данные системы:

- **tickets** — обращения с AI-классификацией и черновиками
- **messages** — история переписки (входящие и исходящие)
- **attachments** — метаданные вложений
- **knowledge_base** — статьи базы знаний с тегами
- **categories** — справочник категорий обращений
- **users** — операторы системы
- **ai_logs** — лог всех действий AI (для отладки и аналитики)

#### Как компоненты взаимодействуют между собой

Описываю словами, потому что это проще понять, чем по схеме.

Фронт общается только с бэкендом. Все запросы идут через REST API: получить список тикетов, получить детали, отправить ответ, поиск по базе знаний, экспорт.

Бэкенд — центральный хаб. Он общается со всеми:

- С почтовым сервером через IMAP (забирает письма) и SMTP (отправляет ответы)
- С ML-сервисом через HTTP (отправляет текст на классификацию и генерацию)
- С PostgreSQL через SQLAlchemy (читает и пишет все данные)
- С фронтом через REST API (отдаёт данные, принимает действия оператора)

ML-сервис изолирован. Он получает запросы только от бэкенда, обрабатывает текст и возвращает результат. Ни с базой, ни с почтой, ни с фронтом он не общается напрямую.

PostgreSQL — пассивное хранилище. К нему обращается только бэкенд.

## 4. Анализ рисков

### Риск 1: ML-модель плохо классифицирует или генерирует слабые ответы

**Вероятность:** высокая. На хакатоне мало времени на сбор данных и fine-tuning.
**Влияние:** высокое — это ядро продукта.
**Как минимизируем:** Уже сделали rule-based классификатор, который работает на ключевых словах. Он показывает приемлемое качество на типовых обращениях. На хакатоне будем надстраивать BERT/RoBERTa поверх. Если модель не дотянет — rule-based останется как рабочий fallback. Для генерации ответов — шаблоны по категориям, которые уже покрывают 80% случаев.

### Риск 2: Не успеваем интегрировать все компоненты

**Вероятность:** средняя.
**Влияние:** высокое — если не склеим, демо не покажем.
**Как минимизируем:** API-контракты между всеми сервисами зафиксированы заранее. Фронт работает на моках и не зависит от бэкенда. Бэкенд работает без ML-сервиса (есть дефолтные значения). ML-сервис работает автономно. На хакатоне просто соединяем — меняем URL и убираем моки.

### Риск 3: Проблемы с доступом к почте на площадке

**Вероятность:** средняя. IMAP/SMTP могут быть заблокированы, нужны app-пароли, 2FA у почтовых провайдеров.
**Влияние:** среднее — без почты система работает, но не полностью автоматически.
**Как минимизируем:** Предусмотрели мок-режим. Если IMAP не настроен, система работает без опроса почты. Тикеты можно создавать вручную через API или загружать .eml файлы. Для демо этого достаточно.

### Риск 4: ML-модель тормозит, долгий inference

**Вероятность:** средняя. BERT на CPU может обрабатывать запрос несколько секунд.
**Влияние:** среднее — оператор будет ждать.
**Как минимизируем:** Обработка идёт асинхронно в фоне. Тикет создаётся сразу со статусом «новый», через несколько секунд переходит в «AI обработан». Оператор не блокируется. Плюс используем легковесную модель (rubert-tiny2) и кэшируем результаты для похожих запросов.

### Риск 5: Docker не работает на площадке или проблемы с окружением

**Вероятность:** низкая.
**Влияние:** высокое — не можем развернуть систему.
**Как минимизируем:** docker-compose протестирован заранее, поднимается одной командой. Запасной план — запуск сервисов напрямую: `uvicorn` для бэкенда и ML, `npm run dev` для фронта, PostgreSQL через Docker или облачный сервис.

### Риск 6: Конфликты в коде при параллельной работе команды

**Вероятность:** низкая.
**Влияние:** среднее — потеря времени на разрешение.
**Как минимизируем:** Проект чётко разделён на модули. Frontend, backend, ML-сервис — это разные папки с разными людьми. Работаем в feature-ветках, мержим в main через PR. Контракты между сервисами зафиксированы — каждый работает автономно.

## 5. Текущее состояние разработки

### Что уже готово

**Backend (FastAPI):**

- Каркас приложения с настроенным подключением к PostgreSQL (async SQLAlchemy + asyncpg)
- Модели данных: Ticket, Message, Attachment, KnowledgeBase, Category, User
- REST API эндпоинты: CRUD тикетов, история переписки, поиск по базе знаний, экспорт в XLSX
- Сервис отправки почты (SMTP, с мок-режимом)
- Сервис-клиент для вызова ML-сервиса
- Фоновая задача опроса почты (IMAP, с мок-режимом)
- CORS настроен для работы с фронтом

**Frontend (React + TypeScript + Vite):**

- Тёмная тема в стиле JARVIS
- Боковое меню с навигацией
- Дашборд со статистикой по статусам, приоритетам, тональностям
- Таблица обращений с фильтрами по статусу и приоритету, поиском по тексту
- Автообновление таблицы каждые 5 секунд
- Боковая панель с деталями тикета, историей переписки, метаданными AI
- Редактор ответа с кнопкой «Вставить AI-черновик»
- Отправка ответа с обновлением статуса
- Экспорт данных
- Страница поиска по базе знаний
- **Всё работает на моках** — переключение на реальный API в одну строку

**ML Service (FastAPI):**

- Rule-based классификатор по ключевым словам (категория, приоритет, тональность)
- Шаблонный генератор ответов по категориям
- Два эндпоинта: `/classify` и `/generate`
- Готов к замене на BERT/RoBERTa на хакатоне

**База данных (PostgreSQL):**

- Полная схема: 7 таблиц с индексами и constraints
- Начальные данные: 5 категорий, 2 оператора
- Миграция через init.sql

**Инфраструктура:**

- docker-compose с 4 сервисами (frontend, backend, ml_service, postgres)
- Dockerfile для каждого сервиса
- nginx конфиг для фронта с проксированием API
- .env.example с описанием всех переменных

### Что будем делать на хакатоне

- Подключить BERT/RoBERTa для классификации вместо rule-based
- Реализовать RAG (Retrieval-Augmented Generation) для генерации ответов на основе базы знаний
- Подключить реальный IMAP-опрос почты
- Добавить WebSocket для мгновенного обновления таблицы
- Наполнить базу знаний реальными статьями
- Добавить аналитику и графики на дашборд
- Полировка UI и UX
