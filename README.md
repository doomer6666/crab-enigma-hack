# Jarvis AI

## Идейная реализация решения

### Пользовательский путь

Рассматриваем сценарий работы оператора технической поддержки с системой.

#### Шаг 1. Получение письма
Система автоматически опрашивает почтовый ящик технической поддержки через IMAP.  
Новое письмо сохраняется в базе данных как тикет со статусом «Новый».

#### Шаг 2. Автоматическая обработка AI
После сохранения письма backend передаёт текст обращения в ML-сервис.  
ML-сервис выполняет:
- классификацию обращения (категория),
- определение приоритета,
- анализ тональности,
- формирование черновика ответа.

Результаты сохраняются в базе данных и привязываются к тикету.

#### Шаг 3. Отображение в веб-интерфейсе
Оператор открывает веб-интерфейс и видит таблицу обращений.  
Для каждого тикета отображаются:
- тема письма,
- отправитель,
- категория,
- приоритет,
- тональность,
- уверенность модели,
- текущий статус.

Таблица обновляется автоматически.

#### Шаг 4. Работа с тикетом
Оператор открывает карточку обращения и видит:
- полный текст письма,
- историю переписки,
- результаты анализа AI,
- сгенерированный черновик ответа.

Оператор может:
- использовать черновик без изменений,
- отредактировать его,
- написать ответ вручную.

#### Шаг 5. Отправка ответа
После подтверждения ответа:
- backend отправляет письмо клиенту через SMTP,
- сообщение сохраняется в истории переписки,
- статус тикета изменяется на «Решён».

#### Сводная схема пользовательского пути

| Этап | Действие системы | Действие оператора |
|------|------------------|-------------------|
| 1 | Получение письма через IMAP | — |
| 2 | Классификация и генерация ответа | — |
| 3 | Отображение тикета в таблице | Просмотр списка обращений |
| 4 | Загрузка деталей тикета | Анализ и редактирование ответа |
| 5 | Отправка письма через SMTP | Подтверждение отправки |


### Архитектура системы

Проект реализуется как набор взаимосвязанных сервисов.

#### Компоненты системы

**1. Почтовый модуль (IMAP/SMTP)**  
- IMAP — получение входящих писем.  
- SMTP — отправка ответов клиентам.

**2. Backend (FastAPI)**  
Центральный компонент системы. Отвечает за:
- получение и парсинг писем,
- взаимодействие с ML-сервисом,
- работу с базой данных,
- предоставление REST API для фронтенда,
- изменение статусов тикетов,
- экспорт данных.

**3. ML-сервис (FastAPI)**  
Отдельный микросервис для обработки текста.  
Предоставляет два эндпоинта:
- `/classify` — возвращает категорию, приоритет, тональность и confidence,
- `/generate` — возвращает черновик ответа.

ML-сервис не имеет прямого доступа к базе данных и взаимодействует только с backend.

**4. База данных (PostgreSQL)**  
Хранит:
- тикеты,
- сообщения переписки,
- результаты анализа AI,
- статьи базы знаний,
- справочники категорий.

**5. Frontend (React)**  
Веб-интерфейс оператора.  
Взаимодействует только с backend через REST API.


### Взаимодействие компонентов

1. IMAP → Backend  
   Backend получает письмо и создаёт тикет.

2. Backend → ML-service  
   Передаёт текст письма для анализа и генерации ответа.

3. ML-service → Backend  
   Возвращает результаты обработки.

4. Backend → PostgreSQL  
   Сохраняет тикет, метаданные и черновик ответа.

5. Frontend → Backend  
   Запрашивает список тикетов и детали обращения.

6. Backend → SMTP  
   Отправляет подтверждённый оператором ответ клиенту.

Таким образом, backend выступает центральным узлом системы и обеспечивает координацию всех компонентов.


## Анализ рисков

### Риск 1. Низкое качество работы ML-модели
В условиях ограниченного времени возможно недостаточное качество классификации и генерации ответа.

**Минимизация:**
- использование rule-based логики как fallback,
- применение шаблонов ответов,
- постепенное улучшение модели на основе тестовых данных.


### Риск 2. Неполная интеграция сервисов
При параллельной разработке возможно несоответствие API или ошибки взаимодействия между сервисами.

**Минимизация:**
- заранее зафиксированные API-контракты,
- изоляция сервисов,
- возможность автономного запуска каждого компонента.


### Риск 3. Проблемы с доступом к почтовым серверам
Ограничения сети или настройки безопасности могут затруднить подключение к IMAP/SMTP.

**Минимизация:**
- наличие мок-режима,
- возможность создания тикетов вручную для демонстрации.


### Риск 4. Производительность ML-сервиса
Обработка текста может занимать значительное время при использовании моделей на CPU.

**Минимизация:**
- асинхронная обработка,
- использование облегчённых моделей,
- кэширование повторяющихся запросов.


### Риск 5. Проблемы с окружением на площадке
Сбои в Docker или различия в среде выполнения могут повлиять на запуск системы.

**Минимизация:**
- предварительное тестирование docker-compose,
- наличие инструкции по локальному запуску сервисов.
